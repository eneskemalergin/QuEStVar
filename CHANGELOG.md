## [0.1.0] - 2026-05-17

Initial release of QuEStVar.

### Statistical engine

- Paired equivalence and difference testing using TOST and Welch's t-test. Both tests run in a single vectorized pass over the full feature matrix, with no Python loops over individual features. Paired samples are supported via `is_paired=True`.
- CV-based quality filter runs before any statistical test. Features with a coefficient of variation above the threshold in either condition are excluded and reported separately in `TestResults.info`.
- Six multiple testing correction methods are available: BH-FDR, Holm, Bonferroni, Hochberg, BY, and Storey q-value. The correction method is set once on `TestConfig` and applied uniformly to all p-value series.
- Results are classified into three statuses: `+1` for features whose effect falls inside the equivalence boundary with a significant equivalence test, `-1` for features outside the difference boundary with a significant difference test, and `0` for everything else.

### Python API

- `QuestVar` is the main analysis class. It accepts a `TestConfig` dataclass, a plain dict, or keyword arguments directly. Configuration can also be loaded from a YAML file via `QuestVar.from_yaml(path)`.
- `QuestVar.test()` runs a single pairwise comparison and returns a `TestResults` object. `QuestVar.compare_all_pairs()` accepts a condition map and runs every pairwise combination in one call.
- `qv.test()` is a module-level shortcut for one-off tests without instantiating `QuestVar` explicitly.
- `TestResults` holds the per-feature result table alongside the config and CV filter metadata. It exposes `.summary()` for a quick text summary, `.save()` for writing to parquet, CSV, or TSV with a companion `.meta.json`, `.load()` for round-trip recovery, and `.plot()` for the summary figure.

### Power analysis

- `run_power_analysis()` sweeps over a grid of equivalence boundaries, replicate counts, CV levels, and CV thresholds to estimate statistical power and the Stable Equivalence Index (SEI). Each design point is evaluated over multiple Monte Carlo iterations.
- Synthetic data is generated by `simulate_data()` using a log-normal intensity model with gamma-distributed per-feature CV, matching the distributional assumptions of typical proteomics data.
- Design points are evaluated in parallel via `multiprocessing.Pool`. The number of workers is controlled by the `n_jobs` argument.
- `PowerResults` holds the aggregated design grid and per-run metrics, with `.summary()`, `.save()`, and `.plot()` methods matching the `TestResults` interface.

### Plotting

- Antler's plot displays signed `-log10(p)` on the y axis against `log2FC` on the x axis. Equivalence and difference threshold lines are drawn in fixed semantic colors (blue for equivalence bounds, red for difference bounds) defined by `PlotConfig`.
- An eight-panel summary figure is produced by `results.plot()`, covering the Antler's plot, p-value distributions for both tests, an MA plot, a status count bar chart, the CV filter exclusion matrix, and sample size comparison.
- A power profile plot shows SEI and power curves across the swept design grid.
- All visual parameters (colors, font sizes, spine styles, grid, annotation behavior) are controlled by a single `PlotConfig` dataclass so that changing one setting propagates consistently to every figure.
- Protein-level annotation is opt-in: pass `protein_ids` for a specific list or `top_n` to label the most significant features per category.

### CLI

- `questvar test` runs a pairwise equivalence test from the command line. Input files can be parquet, CSV, or TSV. A YAML config file can be passed with `--config`, and any individual parameter can be overridden with a dedicated flag.
- `questvar power` runs the power analysis sweep. The design grid can be specified via comma-separated CLI arguments or through a YAML config file.
- `questvar plot` generates antlers or power profile figures from previously saved result files.

### Project

- Source layout with `src/questvar/`, `pyproject.toml`, ruff for linting and formatting, mypy for type checking, and pytest for the test suite.
